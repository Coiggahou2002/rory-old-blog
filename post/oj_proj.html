5个月，我从一个完整项目学到了什么？

蔡嘉豪
2022-02-06
项目经验
前端开发, 经验, 随笔

零、前言

好久没写文章了，今天这期文章，想跟各位计算机专业的新手伙伴们聊一个老生常见的话题。

为什么大家总说要通过做项目来学习？

想想当初的我，也像各位小朋友一样，心里充满了问号：

  • 大家说的“开发”、“做项目”到底是在干啥？
  • 我到底需要学哪些东西、怎么学、学到什么程度，才能做成一个什么样的项目？
  • 什么框架、技术栈、设计模式，他们说的术语是什么，为什么我一个都听不懂？
  • 做开发是不是很 low？是不是只要有算法和科研才有技术含量？

今天这一篇文章，先跟大家聊一下，从零开始，完成第一个项目，为我带来了什么收获？

预计后面再写 1~2 篇，分享一些新手做项目的指南和有用的技巧

注：由于我目前耕耘的领域是 Web 开发，所以今天我谈到的分享也好、心得也好，都只能保证适用于 Web 开发的领域

从去年 7 月份开始（准确来说是 9 月份，因为 7 月和 8 月主要是在学习）到今年年底，大概 4 个月左右的时间，我都在参与学校某实验室的一个 Web 项目，准确来说是一个网站（目前还没正式投入使用）

下面是项目的部分截图：

我在其中，负责前端部分的工作，从启动项目到部署，前端绝大部分的工作和代码都是我负责的，于我自己而言，收获很大。

其实在写到这里的时候，我在很努力地回忆，半年前的我，是什么样子的？

我们都知道知识的诅咒——意思是指，一个已经掌握了知识的人，是很难去想像，那些没有掌握知识的人，他们脑子里的想法，是什么样子的。

我尽可能地去回忆，翻相册，翻朋友圈，翻 GitHub 提交记录，希望能够想起半年前的我，一个零项目经验的新手开发者的想法和疑惑，顺便梳理一下这半年来的心路历程，供自己留一份回忆，如果有幸能够帮到读者，那就再好不过了。

一、我的项目是哪来的？

我是 2020 年 9 月入的学，计算机系科班生，从 2020 年 9 月到 2021 年 5 月期间，也就是入学 8 个月左右吧，活动参与了不少，学业也有点忙，力扣也没少刷，乱七八糟的什么博客、公众号都折腾过，但是 8 个月的时间里，一个项目都没做过，项目经验为零。

那时候的我，是什么水平呢？大伙听我慢慢讲：

我那个时候，总是会听到大佬/学长们说，什么前端开发、后端开发、客户端开发，各种名词都听得耳熟能详，但是一个都不会，一个都没了解过，最多只会“报菜名”。

啥叫“报菜名”呢？就是跟别人吹牛的时候，能说一两句

“诶我知道那啥，后台开发一般用 Java 和 C++ 嘛……”

“前端开发啊，不就是三大件嘛，HTML + CSS + JavaScript”

除此之外，啥都不会。是的，你没看错，除了吹牛报菜名，啥都不会。

菜得要命的我，每天都在想：

怎么办呀？以后找工作也要项目经验呀…

我啥时候能找个项目做啊？做点啥？要学啥才能做？不知道啊…

后来，通过一些研究生学长得知，学校实验室一般时不时会有些项目，需要学生来当免费劳动力，我就厚着脸皮去找了一位教授，问他能不能给个机会我到实验室里干活，当时人家问我，你学分绩排名多少名？我说了下，可能没达到他的要求，教授说：“你先回去好好学习”

哈哈，当时心态有点崩…

后来，也许我在这位教授的实验课上表现比较好，他又找到我说：“咱现在有一个 xx 项目，大概需要这样这样的功能，你有兴趣参与不？”

俺说好，于是教授给让他实验室的一位研一学长来负责带领和组织整个项目的开发

这位学长是保研选手，曾在腾讯后端开发岗实习，是我入门开发的贵人和领路人，我一般都叫他“灿哥”，我写这篇文章的目的之一，咱就是说，也是希望，向灿哥表达一个大大的感谢 ♥

当时这个项目分成了两个 team，分别负责前端和后端开发，各有 3~4 人，灿哥问我想做前端还是后端，我由于自己本来比较喜欢视觉设计，喜欢视图层的东西，就选择了前端开发。

当时我特别兴奋，但同时也有些担心，担心自己无法完成项目的需求；其实现在回头来看，担心都是多余的，朋友们，做事不要畏畏缩缩，撸起袖子直接干就对了！

二、艰难的起步

学习清单

6 月份的时候，灿哥很贴心地写了一份学习指南，告诉我们做项目需要学习哪些预备知识，以及给出了对应的学习资源链接。

前端部分大概包括：

  • HTML
  • CSS
  • JavaScript
  • Vue 基础
  • Element UI 组件库

后端部分包括：

  • Java 基础
  • 数据库基础
  • Linux 基础
  • Spring 和 Spring Boot 框架
  • MyBatis 框架

基础工具和知识：

  • Xshell (用于控制远程服务器)
  • Xftp (用于向服务器上传文件)
  • Postman (网络请求测试)
  • Git (版本控制 + 协作开发)
  • 计算机网络 应用层 + 传输层

前端小组的人需要掌握前端部分，后端小组的人掌握后端部分，而基础工具知识是大家都需要掌握的。

当时我刚好结束考试月，时间比较充裕，就打算把前端、后端都一起学了。

其实当时主要学习重心还是集中在后端，因为我大概知道前端三大件是在干啥，HTML 负责页面结构，CSS 负责样式，JS 负责逻辑和动效，心里有个概念，但是对于后端，了解程度几乎为零，所以希望多学习一些

学习过程

这部分就比较艰难了，因为上面这份清单，对于当时的我来讲，需要接触和学习的新概念、新知识特别多，每天都有海量的概念和知识涌入。

后端部分

先说说后端部分吧，咱的组长说，服务端应用一般跑在 Linux 系统上，于是我就费了很大劲去装了个 Ubuntu 20.04 系统，强迫自己在这个系统上做开发，以便熟悉它。

这时候蛋疼的事情就来了：

常用的软件你得安装吧？你上网搜索怎么安装，然后学了一大堆命令

常用的文件操作，复制、创建、编辑，你得会吧？又学了一大堆命令

有些软件要配环境变量吧？你得知道环境变量在哪配吧？于是你把 Linux 的系统目录结构学了一遍

上面balabala折腾完了，大概知道这个系统怎么用了。下面轮到数据库了。

我按照清单上的框架名搜了一下，MyBatis 是一个专门负责和数据库打交道的框架，反正我也不知道该从哪学起，就先去看 MyBatis 的教学视频——才看了两集，就被告知：学这个之前，需要有 JDBC 基础，以及需要在本地部署好 MySQL 服务

经过一番折腾之后，我先在本地装好了 MySQL，启动了服务，然后马上跑去 B 站学 JDBC (Java Database Connectivity, Java 为连接数据库提供的一套 API)，呼啦呼啦学了一通，发现数据库是连接上了，但是操作数据库的过程中笨手笨脚，不得劲儿，为啥呢？

SQL 语句不熟呗，然后我又跑去学 SQL 语句，学着学着，又发现，数据库表、关系、字段、联结，这些概念，我也不熟，然后又去搞了本电子书恶补了一下……

数据库的基本东西学差不多了，接下来就开始学传说中的 Spring 框架，这里头的学问，可就多了。

随便点开一些讲解 Spring 框架的文章或者视频，都会首先给你讲一堆概念

其中最典型的当属 IoC (Inversion of Control, 控制反转) 和 AOP (Aspect Oriented Programming, 面向切面编程)

先别说这些概念了，刚开始学没多久，我就发现我缺少了一些面向对象的知识（类，对象，接口，实现，封装，继承，多态），灰头土脸地跑回去恶补了一通

回来之后，开始啃有关 IoC 的知识，当时看了好多个版本的讲解视频，反复翻了好多篇文章，记了很多笔记，终于理解了这个知识点（现在回头看看，这个概念对于没有开发经验的新手来讲，确实是非常难啃的）

后续的学习还有 Spring MVC 和 Spring Boot，这部分因为当时时间有点紧，我跳过了很多东西，就不展开说了。

前端部分

前端部分的学习倒没有那么曲折，当时我记得我是在一个叫 freeCodeCamp 的网站学的，这个网站里给你规划好了学习路线，课程分成了很多小节，边写边实践，稍微记下笔记，掌握得还是比较快的。

前后端整合

按照原定的前端学习路线，在大致过了一遍 HTML、CSS、JavaScript 之后，我应该去学 Vue 框架，但当时我去看 Vue 的教学视频，看了几节之后就看不下去了，为啥呢？

因为视频一上来，就教你 Vue 的常见指令的用法，各种 v-on，v-bind 什么的，我都不知道这些东西能用来干啥，学习过程味同嚼蜡，我就放弃了。

其实主要是因为我心里一直有个疙瘩：

前端、后端分别都学了那么久，学了那么多知识，但我至今不知道前后端是怎么连接在一起、怎么通信的，我连个像样的小东西都做不出来，实在是有点沮丧。

直到我发现了一个叫做《4 小时上手 Spring Boot + Vue 前后端分离开发》的视频，视频 up 主没有任何废话，上来就带你分别创建 Spring Boot 项目和 Vue CLI 项目，在数据库简单建了个表，然后用 axios 直接开始收发 http 请求，打通前后端，启动了一个包含了基本增删改查操作和路由导航的项目。

看完这里，我有一种感觉：

闭关静修久矣，今君一指，吾任督二脉通也

无论是前端、后端，之前学的所有零散的知识点，全都串起来了！

当然，同时又发现自己还需要学习以下知识：

  • HTTP 协议
  • Vue CLI 脚手架
  • Vue 框架用法
  • Vue Router 路由配置
  • Element UI 组件库
  • …

学习体会

其实，通过前面的长篇流水账叙述，大家可以看到，我的学习过程非常的艰辛、曲折。

那段时间 (暑假七八月份) 里，每天的日常就是电脑分成 2 个屏，左边写代码/看视频/看文章，右边记笔记，经常不知不觉学到凌晨 2 点、3 点，甚至有时候 4 点都还没睡。

现在回头再看那段时间的学习过程，其实所学的东西和学习的路径大致可以用下面这张图来概括：

在学 A 的过程中，发现自己 B 也不会，然后钻进去学习 B；

学习 B 的时候，又发现还需要先学 C，又钻进去学 C…

一层一层地往里钻，当觉得自己当前所在层学得差不多了，就回到上一层继续学

如此反复，当回到最顶层的时候，说明这个板块掌握得差不多了

用计算机的术语来讲，这个过程就像递归，当然，更恰当的说法是深度优先搜索。

当然了，这个过程也可以理解为，对一棵多叉知识树的前序遍历。

为什么会这样呢？

因为我的学习方向不是自底向上，而是自顶向下的

当我对将要学习的东西缺乏一个整体的认知

或者说，我根本不知道我需要学习哪些东西的时候

我会选择从最顶层开始学

这样一来，我需要学习的知识就会在路上一个个浮现

这时，学习路径就会整体呈现为一个向下的树状结构

需要我一层一层往下潜，再一层一层往上浮起来

辛苦归辛苦，也很考验人的耐心

但只要坚持下来，收获还是很大的

这一整个过程就是入门 Web 开发最艰难的过程，也是从 0 到 1 的过程，它之所以难，不是难在需要学习的知识点多而繁杂，而是难在我们根本不知道要学什么，这时候，这种目标驱动、自顶向下的方法，我认为是比较合适的。

三、登录认证模块——第二个瓶颈

最艰难的一步，或者说，学习 Web 开发路上的第一个瓶颈，已经迈过去了。

没过多久，我又迎来了第二个挑战——灿哥让我全权负责项目的登录认证模块的前端部分，并与一位负责后端的大三学长共同完成这个模块。

其实这个模块无论是前端部分还是后端部分，都是有一定难度的，刚入门就担此大任，估计又免不了一番爆肝了…

当然，这个模块有一个很明显的好处，那就是网上有很多现成的方案可以参考，在调研过程中，我主要做了以下几件事：

  • 搜索查询主流的解决方案
  • 了解/学习这些方案涉及到的技术、用到的框架和相关知识点
  • 找开源项目的有关示例，参考别人的设计思路

记得为了补充需要的 HTTP 协议知识，我用了一天左右的时间刷完了《图解 HTTP》这本书，还看了《网络是怎样连接的》的其中一部分

做登录模块，涉及到密码，肯定少不了加密，需要必要的密码学常识，为了搞清楚 MD5, SHA-256, RSA, AES 这些到底都是什么东西，它们之间有什么关系，我又用两三天时间过了一遍《图解密码技术》

为了搞明白 Cookie, Session, Token, JWT 等各种各样的鉴权机制/工具，我翻了很多很多篇文章（其实网上很多地方写得都不太准确，后来找了一些英文资料才搞明白）

搞明白鉴权的基本原理和方案之后，下一步就需要将它落地了。

项目组里决定在后端集成 Spring Security 框架来做这个事情，我又特地去了解了一下这个框架相关的知识，然后找了一个叫 springboot-rbac-security 的 demo，结合前端这边做鉴权的思路，梳理出了一个总的方案，在当时的项目周会上做了登录认证授权模块的方案详解和分享。

（当时还认真做了好多页 PPT，花了很长时间，画了一个总流程图）

这一整个方案，从调研到确定到落地，无论是我还是后端的大兄弟，都踩了不少坑，最后才做出来。

还有很多踩坑的细节，我就不一一展开了。

我想说的是，在确定的需求面前，知识的扩充是自然而然的：

当你的业务逻辑中需要加密，你就会去了解加密算法

当你需要鉴权，你会去了解 Token 令牌

当你需要在无状态协议下保存会话信息，你会去了解 Cookie, Session

当你在想，服务端将会话信息存在什么地方合适，你会去了解 Redis

当你的前端页面需要配合后端做鉴权，控制哪些页面用户能访问、哪些不能访问，你需要了解 Vue 路由拦截器和 Axios 响应拦截器

当你需要在前端存储全局信息，你需要了解 状态管理库 Vuex

……

总而言之，在面临一个需求时，只要你认真调研，用心去做，在遇到不懂的概念时不要得过且过，花点时间补充必要知识，对自己的方案的代码负责，最终的收获一定不会少。

四、需求和学习的深入

项目的基础框架和较难的模块都做好之后，在后续的项目进程中，日常开发流程都是下面这样：

  1. 灿哥分配功能需求
  2. 前端和后端一起讨论，设计方案
  3. 前端负责页面设计 + 布局 + 交互逻辑 + 数据展示
  4. 后端负责和数据库打交道，提供数据 API 接口
  5. 前端、后端联调

在这个过程中，陆陆续续做了这些模块/功能：

  • 用户头像上传/修改
  • 修改密码/忘记密码
  • 绑定邮箱
  • Markdown 文章编辑/展示
  • 评论模块
  • 点赞功能
  • 悬停自动搜索
  • 做题/判题模块
  • ……

在代码量逐渐增大的过程中，增强可维护性的需求就会自然显现

多处都用到的逻辑，需要抽离到公共的工具类中；

一个UI/功能/逻辑完整的可复用单元，需要被封装成 Vue 组件；

组件之间出现了父子/兄弟/祖孙关系，需要传递数据，就需要了解组件通信的各种方法，包括 props/emit, $refs, eventBus, provide/inject…

要做 Markdown 编辑/渲染，要找稳定的开源的轮子/插件，比如字节的 @bytemd/vue；

要做代码编辑器，找到开源的 mavonEditor 之后，还要踩一遍它布局 bug 的坑；

要在用户切换页面之后保留之前编辑的内容，需要做路由组件缓存；

要做头像上传功能，就要跟云存储商的 API 和 SDK 打交道，这个功能还因为云存储商的 CDN 延迟出过一点小问题，当时因为要向老大梳理和汇报这个事情，又写了一篇文章…

以上都是日常开发过程中，边踩坑边学习的过程

但是，下面这件事情，才让我真正地发现了新大陆：

在数字逻辑考试那天，前端这边一个组员写出了一个莫名其妙的 bug，我将这个 bug 复现之后，在网上没有找到相关的资料

我的直觉告诉我，这与 Vue 框架的响应式原理有关

于是我抛下手头上正在复习的数字逻辑 PPT

纵身一跃，跳进了 Vue 框架 2.6.14 版本的源码中寻找答案

最终发现，答案就在这段代码中

这次机缘巧合，让我对 Vue 的响应式原理有了更进一步的了解

更重要的是，它开启了我阅读框架源码的不归路……

五、部署阶段——另一番风景

前面吭哧吭哧做了那么久，项目做差不多了，就要考虑部署和上线的问题了

因为我们是前后端分离开发，也就是说，前端和后端是分开部署的，最后可能分别在两台不同的服务器上，也有可能在一台服务器的两个不同端口上

也就是说，我只要负责把前端部分部署好就行了

第一次部署的我，天真无邪，编译打包之后，直接扔上了我自己的阿里云服务器

结果发现，首次访问的时候，居然加载了 50 秒左右才进入主页！

你能想像吗？打开一个网页需要“转圈圈”50秒才能进入！

我被吓得不轻，急忙去 Chrome 控制台的网络面板看看到底传输了啥东西

经过一番排查和 Google，发现其实是编译打包的时候，Webpack 打包的体积太大了，导致页面首次加载的时候需要在有限的网络带宽上传输大量臃肿的 js 文件，导致速度异常缓慢

所以接下来的任务很简单，就是要想尽一切办法，减少 Webpack 的 bundle size：

1.用 Webpack-Bundle-Analyzer 分析 bundle 结构

2.将项目依赖的几个核心公共库和组件库改为 CDN 引入

之前在开发环境下，这些核心库和组件库（如 Vue，Vuex，Vuetify）都是通过 npm 包安装的，所以它们最终全部都会被打包进 Webpack 的 bundle 里面，现在将它们从依赖列表中移除，全部改为 CDN 引入，相当于减轻了服务器的传输负担，从多路获取资源，原来的串行传输变为并行传输，速度快了不少

3.路由组件全部改成异步加载

异步加载也称为懒加载，使用懒加载之后，进入首屏只需要加载首页的路由组件，速度又快了不少

4.关闭生产环境的 Source Map

Vue CLI 集成了代码压缩，编译打包时会压缩代码，而 Source Map 是一种存储映射信息的文件，能够将『压缩后代码』映射为『原代码』，它的功能主要是方便开发人员在线上环境直接调试，所以默认也会被打包进 bundle 里，我们只要去掉这些映射文件，就能够进一步减小打包体积

其实上面几点我都是一步一步摸索过来的，并不是一口气完成的

我的首屏加载时间慢慢从 50 秒优化到 49 秒

从 49 秒 优化到 35 秒

20 秒、9 秒、5 秒….

最终达到一个正常情况下 1.5 ~ 3 秒左右的首屏加载时间

最核心的访问速度问题解决了之后，还有一些比较杂碎的问题，这里也不展开了：

  • 某些涉及 ajax 请求的操作 / 会频繁触发的数据，视需要做防抖或节流处理
  • 配合后端排查接口响应问题
  • 排查潜在的网络安全问题（未完成）
  • ……
