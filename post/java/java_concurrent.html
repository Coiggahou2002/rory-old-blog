Java 并发编程笔记

Coiggahou
2022-08-31
Java, 并发编程, 操作系统

线程的基本特点

每个线程独有的：PC，执行栈，线程本地变量

多个线程共享的：堆区，方法区

创建线程的方法

  • 继承 Thread 类重写 run()
  • 实现 Runnable 接口
  • 使用 FutureTask<T>（将要执行的任务包装成 Callable，放入 FutureTask，其中 T 是线程返回值类型）

线程的类型

Java 中线程的类型有两种：

  • 用户线程
  • 守护线程

线程的行为

某个线程调用 lockobj.wait()，则该线程挂起，阻塞自己（直到被其他线程唤醒），共享对象 lockobj 的锁会被释放（只会释放 lockobj 的锁，不会释放持有的其他锁）

某个线程调用 Thread.sleep(t)，则该线程睡眠 t 秒，在睡眠过程中不争夺 CPU 时间片，但是不会释放自己持有的锁

某个线程调用 lockobj.notify()，则会选中 lockobj 的阻塞集合的其中一个挂起线程（随机选择），唤醒该线程

某个线程调用 lockobj.notifyAll()，则会选唤醒 lockobj 的阻塞集合中的所有挂起线程

假设一个线程之前调用过 lockobj.wait() 而进入了挂起状态，然后另外一个线程调用 lockobj.notify() 唤醒了它，需要注意的是，它被唤醒后并不会立刻继续执行，它必须参与竞争，拿到 lockobj 的锁之后，才能够继续执行

一个线程调用 Thread.yield()，意思是告诉 CPU，我事情做完了，剩下的时间片不想用了，我现在进入就绪状态（注意是就绪而不是挂起），你可以开始下一轮调度了

某个线程 K 对线程 A, B, C 调用了 join() 方法，那么 K 会阻塞自己，等待 A, B, C 完成

对某个线程调用 threadA.interrupt() 可以将 threadA 的中断标志设置为 true

对某个线程调用 threadA.isInterrupted() 可以得到 threadA 的中断标志是否为 true

当前线程调用静态方法 Thread.interrupted()，会返回当前线程自己的中断标志是否为 true，如果为 true 还会清除掉中断标志位

从共享对象的角度看待锁？

我们上面多次提到，锁是通过一个中介共享对象 lockobj 实现的

如果我们要实现 wait() 和 notifyAll() 方法，我们的 lockobj 至少需要维护的数据：

  • 谁调用过 lockobj.wait() ？（阻塞线程等待室）
  • 我的锁只有一个，这个锁当前被谁持有？

共享变量问题

多线程共享变量时，同时读写变量，就会出现线程安全问题

线程安全问题：指当多个线程同时读写一个共享资源并且没有任何同步措施时，导致出现脏数据或者其他不可预见的结果的问题

线程可以有属于自己的本地变量，可以通过 ThreadLocal<T> 来创建

每个线程可以有很多个属于自己的本地变量，这些本地变量实际上存在这个线程结构里的一个叫做 ThreadLocalMap 的结构中，这个结构其实就是一个 HashMap

注意：子线程无法获取父线程的 ThreadLocal 变量值，如果有这个需求，需要使用 InheritableThreadLocal 类型

生产者消费者问题

生产者 $P$ 和消费者 $C$ 同时盯着产品队列 $Q$

当 $P$ 拿到 $Q$ 的锁时：

  • 如果队列满，调用 Q.wait() 挂起自己，并释放锁
  • 否则，往队列里加东西，然后调用 Q.notifyAll() 唤醒 $C$

当 $C$ 拿到 $Q$ 的锁时：

  • 如果队列空，调用 Q.wait() 挂起自己，并释放锁
  • 否则，从队列中取出东西，然后调用 Q.notifyAll() 唤醒 $P$

线程死锁

死锁产生的四个条件

  1. 互斥（拿在手里的不给你）
  2. 请求并持有（吃着碗里，还要锅里的）
  3. 不可剥夺（用完之前不能让出）
  4. 环路等待（依赖链有环）

如何避免死锁？

核心思路：破坏四个条件其中的一个

只有请求并持有和环路等待是可以被破坏的

具体实践：依靠资源申请的有序性原则

重要关键字

synchronized 关键字

基本行为

被该关键字包括的代码块称为同步代码块，进入同步代码块，会自动获取 obj 对象的锁（如果没获取到，不会进入代码块）

什么时候释放这个锁？

  • 正常退出代码块
  • 同步代码块中途抛异常
  • 同步代码块中调用 obj.wait()

内存语义

进入同步代码块时，把同步代码块中所有使用到的变量从线程的工作内存中清除，直接从主内存中获取这些变量

退出同步代码块时，把在同步代码块中对变量的修改，刷新到主内存

为什么 synchronized 开销大？

因为 Java 的多线程锁和操作系统的锁是一一对应的，进入 synchronized 代码块需要获取锁，而获取锁的过程，需要从用户态切换到内核态，这里涉及到 OS 的上下文切换，所以开销大

volatile 关键字

volatile 关键字用于解决共享变量的内存不可见性（内存不可见性本质上源于 cache 和主存之间的一致性问题，由于缓存的存在，多个线程同时修改某个变量时，变量的最新版本可能并不会被其他线程马上看见）

使用该关键字声明的变量，可以确保对这个变量的更新操作（读/写）会马上刷新到主内存（对其他线程立即可见）

另外，volatile 声明变量，还可以避免处理器进行指令重排：

  • 写 volatile 变量时，可以确保写之前的指令不会被编译器放到写之后
  • 读 volatile 变量时，可以确保读之后的指令不会被编译器放到读之前

个人理解（不一定对）：使用 volatile 关键字声明的变量，相当于，每一处读写该变量的代码，都被包裹上了 synchronized 关键字，成为了很小的一个同步代码块

优缺点：

  • 开销比 synchronized 要小
  • 只保证内存可见性，不保证原子性
  • 只适用于要写入变量的值不依赖当前变量值的情况，例如 a=1 可以，但是 a=a+1 不行

锁

乐观锁与悲观锁

乐观锁：对数据被外界修改持乐观态度（认为数据在一般情况下不会冲突），处理数据的时候不加锁，而是知道提交更新的时候，才会检测数据是否冲突

悲观锁：对数据被外界修改持悲观态度（永远作最坏的打算），总是在处理数据前先对数据加锁

公平锁与非公平锁

公平锁：先到先得

非公平锁：先到不一定先得

（公平锁会带来额外的性能开销）

独占锁与共享锁

独占锁：只能被单个线程持有（是一种悲观锁）

共享锁：可以同时被多个线程持有
