基本意识

  1. this 并不指代函数对象本身，也不指代函数的词法作用域
  2. 在 global scope 中定义的变量，其实就是 global-object 的一个 property

this 是如何被确定的

函数执行上下文

When a function is invoked, an execution context is created. The execution context contains information about:

  • Where the function was called from (call-stack)

  • How the function was invoked

  • What parameters were passed

  • this reference

Call-Site

Call-Site: the location in code where the function is called

函数永远是 object，通过赋值，将 global-object 的 property bar 设置为函数 foo，此时再通过 bar() 调用，那么 this 就是 global-object

不变且重要的观点：a function is always an object, what we are passing back and forth is just the reference of that object, but the Call-Site (or somehow the context) is what determines the this-binding for the function.

调用函数的几种方式

  • 直接通过名字(别名)来调用 foo()

  • 作为对象的 property 调用 obj.foo()

  • 通过 call 调用 foo.call(obj, ...args)

  • 通过 apply 调用 foo.apply(obj, args)

几种 this 绑定行为

default binding

直接通过名称调用的函数，this 绑定为 global-object，在 use strict 模式下，会绑定为 undefined

implicit binding

作为对象的 property 调用的函数，this 绑定为对象引用

explicit binding

通过 call 或 apply 显式地指定 this

hard binding

使得函数的 this 总是某个指定的对象

this 的好处

this 允许函数在不同的上下文对象中被复用（而不是在各种上下文对象中都放一个具有同样功能的函数副本）

其实也可以显式声明上下文

所以，this 的重要作用之一就是：允许隐式地传递上下文对象（是否优雅就见仁见智了）

与 new 构造函数的关系

在平时编码中，我们通常将 new 操作符 + 函数“当作”类似 OOP 语言中的 contructor function 来使用，but what new under the hood should not be recognized as a constructor function

当我们在函数调用前使用 new 关键字时，流程是这样的：

  • an empty object {} is implicitly created

  • this empty object is binded as the context object of the function

  • the object is returned after some of its properties being set by the function

不严谨的地方

上述实现都是 quick-dirty-impl，很多地方没有严谨考虑：

  • 参数合法性

    • context 对象的合法性，如果不合法要默认设置为 window 对象
    • args 参数数组的合法性

  • call 和 apply 需要将函数临时挂到 context 对象上，但是起名字的时候，需要考虑 context 对象恰好已经有同名 property 的情况，如果已经有了，需要再另外生成随机的名字
